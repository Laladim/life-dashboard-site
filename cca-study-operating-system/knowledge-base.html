What the exam actually is

  • Name: Claude Certified Architect - Foundations (CCA-F). The official guide does not state a launch date.
  • Format: all multiple choice, 1 correct answer of 4 options (the 3 wrong ones are "distractors" a candidate with incomplete knowledge might pick). No penalty for guessing, so answer everything. The exact question count and time limit are NOT in the official guide.
  • Score: pass/fail against a minimum standard set by subject-matter experts, reported as a scaled score of 100-1000, 720 to pass. Scaled scoring equates across exam forms of slightly different difficulty.
  • Official practice exam exists: the guide says to complete the official practice exam before the real one (link provided separately). It uses the same scenarios and format and shows explanations after each answer. This is the closest proxy to the real thing, much closer than the Academy course quizzes.
  • Attempts/cost: The official guide does not state retake, cost, or discount policy. Treat those as registration/admin details to verify outside this guide, not exam-study facts.
  • The key structural fact: every question is anchored to a production scenario. The exam picks 4 of the 6 scenarios at random and writes all questions against those. So the unit of study is the scenario, not the topic.
  • What it tests: practitioner judgment and trade-offs, NOT theory and NOT writing code. It shows a situation and asks which design is right.
  • Candidate profile (this is Shela): "a solution architect who designs and implements production applications with Claude." Not a developer. Hands-on experience with the Claude Agent SDK, Claude Code, the Claude API, and MCP.

The 5 domains (with real weights and plain-English meaning)

Teach a newbie the plain-English line first, then the formal name.

#DomainWeightPlain English
1Agentic Architecture & Orchestration27%When you have one AI helper vs a team of them, and how the "manager" hands work to "workers" and recovers when one fails.
2Tool Design & MCP Integration18%How you give the AI buttons to press (tools), so it picks the right one and fails gracefully. MCP is the standard plug for connecting those buttons.
3Claude Code Configuration & Workflows20%How you set up Claude Code for a team: shared instruction files, custom commands, when to let it plan vs just run, and wiring it into automated pipelines.
4Prompt Engineering & Structured Output20%How you ask so you get back clean, machine-readable data (JSON) reliably, with examples, validation, and retries.
5Context Management & Reliability15%How the AI remembers what matters across a long conversation, when to hand off to a human, and how it stays trustworthy.

The spread is flat. Domain 1 leads at 27%, then 3 and 4 at 20% each and 2 at 18%, with 5 at 15%. This is not "study one thing." (Official order, verified Jun 17.)

The 6 production scenarios (official text, plain gloss, and domains)

Drawn 4-of-6 at random. A newbie should be able to say, for each: what is being built, and which decisions it forces.

  1. Customer Support Resolution Agent. Build a support agent on the Claude Agent SDK handling high-ambiguity requests (returns, billing disputes, account issues) via custom MCP tools (get_customer, lookup_order, process_refund, escalate_to_human). Target 80%+ first-contact resolution while knowing when to escalate. Domains: 1, 2, 5.
  2. Code Generation with Claude Code. Use Claude Code for code gen, refactoring, debugging, docs. Custom slash commands, CLAUDE.md config, plan mode vs direct execution. Domains: 3, 5.
  3. Multi-Agent Research System. A coordinator delegates to specialized subagents (one searches web, one analyzes docs, one synthesizes, one writes reports). Produces comprehensive, cited reports with graceful failure handling. Domains: 1, 2, 5.
  4. Developer Productivity with Claude. Build dev-productivity tools on the Claude Agent SDK: explore unfamiliar codebases, understand legacy systems, generate boilerplate, automate repetitive tasks. Uses built-in tools (Read, Write, Bash, Grep, Glob) + MCP servers. Domains: 2, 3, 1.
  5. Claude Code for Continuous Integration. Integrate Claude Code into CI/CD: automated code reviews, test generation, feedback on pull requests. Design prompts that give actionable feedback and minimize false positives. Domains: 3, 4.
  6. Structured Data Extraction. Extract info from unstructured documents, validate with JSON schemas, maintain accuracy, handle edge cases gracefully, integrate downstream. Domains: 4, 5.

Note: scenarios 1, 3, and 4 are built on the Claude Agent SDK (half the exam). The Agent SDK is the biggest gap not deeply taught by the Anthropic Academy courses.

The 5 decisions (the judgment spine)

~80% of questions reduce to one of these. For each, the exam plants a TRAP and rewards a RIGHT instinct.

  1. One agent, or many? Trap: the impressive multi-agent answer. Right: the simplest thing that works. Multi-agent only when tasks are genuinely parallel and specialized. The exam punishes over-engineering.
  2. What can this tool do, and what does it return when it fails? Trap: a broad tool that throws a raw exception. Right: narrow boundary, clear schema, structured error the agent can handle.
  3. How do I force parseable output? Trap: "prompt it more nicely." Right: JSON schema + validation + retry.
  4. What when the model is unsure or context is too long? Trap: trust the model's confidence or the customer's sentiment. Right: explicit deterministic triggers (human request, policy gap, no progress) + summarize state at handoffs. (The exam calls confidence self-ratings and sentiment "unreliable proxies.")
  5. Claude Code: run or plan, and how into CI? Trap: full autonomy everywhere. Right: plan mode for ambiguity/large changes, prompts tuned for actionable feedback and few false positives.

The hard facts (the only things worth rote-learning)

Grouped by decision. These appear as questions; you either know them or you don't.

Orchestration (Domain 1)

  • Only stop_reason == "end_turn" reliably signals completion. Never parse assistant text or use iteration caps as the stop condition.
  • Subagents inherit nothing from the coordinator. Every needed piece of context must be put in the subagent's prompt explicitly.
  • If the coordinator already has the context and the task is small, do it inline. Spawning a subagent is slower and costs more.
  • For guarantees with financial, legal, or safety stakes, use hooks (code), not prompts. Prompts are probabilistic guidance only.
  • Match the pattern to the work: prompt chaining (fixed steps), routing (distinct categories), orchestrator-workers (delegation), dynamic decomposition (investigations where the next step depends on the last finding).
  • Hub-and-spoke: all communication flows through the coordinator for observability and error control.

Tool design & MCP (Domain 2)

  • Tool descriptions are the primary mechanism the model uses to choose a tool. Overlapping descriptions cause misrouting. Fix by renaming to remove overlap, not by adding a classifier.
  • A good tool makes the right action easy and the wrong action hard or impossible. Prefer parameters that match the real domain model; don't make the model reconstruct business rules from loose strings.
  • Errors: return isError: true with structured fields (category, isRetryable, attempted query, partial results, alternatives). Never throw a raw exception. Never return an empty list to signal failure (empty = success with no matches).
  • Classify failures: transient infra (retry inside the tool), validation (structured detail so the agent corrects), business rule (non-retryable + explanation), permission (escalation path), uncertain write (do NOT auto-retry; report uncertainty).
  • tool_choice: "any" forces a tool call; "auto" allows a text reply; a named tool forces that specific one.
  • Use lookup-then-act: a search tool returns candidates, a separate tool acts on an unambiguous ID. For destructive actions, preview-then-execute with a single-use confirmation token.
  • MCP: Tools = actions; Resources = passive reference context (schemas, catalogs, docs); Prompts = reusable templates. Annotations like readOnlyHint are not a security boundary; enforce security in code.

Structured output (Domain 4)

  • tool_use with a JSON Schema is the most reliable way to guarantee valid JSON. It guarantees shape, not source truth or correctness of values.
  • Make fields optional/nullable when the source may not contain them. Required fields encourage fabrication. Add "other"/"unclear" escape hatches for evolving domains.
  • Few-shot examples (2-4 targeted ones) are the most effective way to get consistent, actionable output.
  • On a validation failure, resend: original document + the bad extraction + the exact errors. Blind retry fails when the info is simply absent.
  • Self-check trick: extract both calculated_total and stated_total to detect discrepancies. Normalize in the prompt (dates to ISO 8601, currency to amount+code).
  • Batch API: ~50% cheaper, up to 24h, no latency SLA. Good for overnight reports; never for a blocking pre-merge check.

Context & escalation (Domain 5)

  • Escalation triggers are explicit: customer asks for a human, policy gap/exception, agent cannot make progress, unsafe/uncertain state. Not sentiment, not model confidence self-ratings (poorly calibrated). Don't use simplistic counters like "escalate after 3 failed tools."
  • Progressive summarization destroys numbers, percentages, and dates. Keep a separate structured "case facts" block outside the summarized history, and preserve claim -> source mappings (URL, doc, quote, date).
  • The model sees a request, not your database. You decide what to include each turn. A big context window does not mean every detail is salient (capacity != attention).
  • Don't resume an old transcript with stale tool results; start from a structured summary + fresh lookups.
  • Escalation handoff carries: customer ID, issue type, root cause, relevant records, amounts, actions taken, recommended next action.

Claude Code & CI (Domain 3)

  • Plan mode = workflow gating (explore read-only -> propose plan -> wait for approval). Extended thinking = reasoning quality. Different mechanisms; confusing them is a planted trap.
  • Plan mode for: many-file changes, architectural choices, migrations/breaking changes, unfamiliar codebases, when approval is needed. Direct execution for a single clear low-risk fix.
  • CLAUDE.md hierarchy: project .claude/CLAUDE.md (shared via version control, all contributors) vs user ~/.claude/CLAUDE.md (only you, not shared). Common bug: putting team instructions at user-level so teammates miss them.
  • Share slash commands via .claude/commands/ in the repo. Personal commands go in ~/.claude/commands/.
  • Path-specific rules live in .claude/rules/ with YAML paths glob frontmatter; they load only when editing matching files (saves context).
  • CI: use -p/--print for non-interactive mode; --output-format json for structured output. Isolate the review context from the session that wrote the code (a fresh instance finds more). When re-running, report only new/unfixed issues.
  • Grep searches text inside files; Glob searches filenames/paths. Don't use filename search for code references.

The build-and-break exercise (the differentiator)

The single highest-leverage practical task. Doing it once beats months of watching videos, and it doubles as a deployment artifact toward partner status.

Build the smallest possible system on the Claude Agent SDK:

  • a coordinator that delegates to two subagents (one "searches," one "synthesizes"),
  • wired to one custom MCP tool you design yourself with a clean schema and a structured error,
  • with the final answer forced into a JSON schema.

Example tool signature (teaches boundary + structured error + no raw crash):

get_record(id: string) -> { found: bool, data: {...} | null, error: "not_found" | "backend_down" | null }

Then break it on purpose. Feed a non-existent ID. Make the backend "go down." Ask something it cannot answer. Watch it fall over, then add the escalation path and a fallback so it degrades gracefully instead of hallucinating. Shela already has this shape in her newsletter pipeline; strip that to a skeleton and rebuild it clean on the SDK rather than starting blank.

Exam-room tactic: delete 3 wrong answers fast

Confirmed against 12 real sample answer keys. The losing options are almost always one of these four shapes:

  1. The one that over-engineers (multi-agent when one agent would do; a classifier when a rename would do).
  2. The one that ignores failure (no structured error, no escalation, no fallback; empty list for an error).
  3. The one that trusts the model to behave (no schema, no validation; relies on sentiment or confidence self-ratings).
  4. The one that gives full autonomy where plan mode or a human gate or code-level enforcement clearly belongs.

What survives is usually the answer. The exam rewards the boring, safe, well-bounded design.

Task-statement-level syllabus (the real topic + keyword checklist)

The exam guide "lists the domains and task statements tested." These are the reproduced task statements, one cluster per domain. Teach these as the actual study checklist; each bullet is a topic, and the inline code terms are the exact keywords a candidate must recognize. 30 task areas total.

Domain 1 - Agentic Architecture & Orchestration (27%) - 7 task areas

  • 1.1 Designing agentic loops. Agent loop lifecycle: send request, check stop_reason ("tool_use" vs "end_turn"), execute tools, append results, repeat. Model-driven decisions vs hard-coded trees. Anti-patterns: parsing assistant text for completion; arbitrary iteration limits as the stop.
  • 1.2 Multi-agent orchestration (coordinator-subagent). Hub-and-spoke: coordinator owns comms, errors, routing. Subagents have isolated context. Coordinator does decomposition, delegation, aggregation, dynamic selection. Risk: overly narrow decomposition. Split coverage to avoid duplication; iterative-refinement re-routing.
  • 1.3 Subagent calls, context passing, spawning. Task tool spawns subagents; coordinator allowedTools must include "Task". Explicit context passing (subagents inherit nothing). AgentDefinition (descriptions, system prompts, tool constraints). fork_session. Parallel subagents via multiple Task calls in one turn. Prompt with goals + quality criteria, not step-by-step.
  • 1.4 Multi-step workflows, enforcement & handoff. Programmatic enforcement (hooks/preconditions) vs prompt guidance. Deterministic guarantees (verify identity before a financial op). Preconditions block downstream (block process_refund until get_customer returns a verified ID). Structured escalation handoff.
  • 1.5 Agent SDK hooks. PostToolUse to intercept/normalize results (Unix to ISO 8601, status codes). Interception hooks block policy violations (block refunds over $500) + redirect to escalation. Hooks = deterministic; prompts = probabilistic.
  • 1.6 Task decomposition. Fixed pipelines (prompt chaining) vs dynamic adaptive decomposition. Per-file analysis + integration pass. Map structure first, then prioritized plan.
  • 1.7 Session state, resume, fork. --resume <session-name>; fork_session branches; inform agent of file changes on resume; a new session + structured summary can beat resuming with stale results.

Domain 2 - Tool Design & MCP Integration (18%) - 5 task areas

  • 2.1 Tool interfaces & descriptions. Descriptions are the primary tool-selection mechanism; include input formats, example queries, edge cases, boundaries. Overlap causes misrouting; rename to remove overlap (analyze_content to extract_web_results); split general tools into specialized.
  • 2.2 Structured error responses (MCP). isError flag. Error types: transient / validation / business / permission. Structured metadata errorCategory, isRetryable, message. retryable: false for business rules. Local recovery in subagents. Access failure vs valid empty result.
  • 2.3 Tool allocation & tool_choice. Too many tools (18 vs 4-5) reduces reliability. Scope tools to role + limited cross-role utilities. tool_choice: "auto" / "any" / forced {"type":"tool","name":...}. Constrained alternatives (fetch_url to load_document).
  • 2.4 Integrating MCP servers. Project .mcp.json (team) vs user ~/.claude.json (experiments). Env-var substitution ${GITHUB_TOKEN}. All connected servers' tools discovered simultaneously. MCP resources as content catalogs (schemas) to cut exploratory calls. Prefer community servers.
  • 2.5 Built-in tools. Grep (content search), Glob (file discovery), Read/Write (full file), Edit (unique-match precise change; fall back to Read+Write if non-unique). Incremental investigation: Grep entry points, then Read flows.

Domain 3 - Claude Code Configuration & Workflows (20%) - 6 task areas

  • 3.1 CLAUDE.md hierarchy, scope, modular. User (~/.claude/CLAUDE.md, not shared), project (.claude/CLAUDE.md or root, shared via VCS), directory-level. @path imports (@./standards/coding-style.md). .claude/rules/ for topic files. Diagnose user-vs-project mistakes.
  • 3.2 Slash commands & skills. Project .claude/commands/ (shared) vs user ~/.claude/commands/. Skills .claude/skills/ with SKILL.md frontmatter: context: fork, allowed-tools, argument-hint. fork isolates verbose output.
  • 3.3 Path-specific rules. .claude/rules/ YAML paths globs; load only when editing matching files; globs like */.test.tsx; prefer over directory CLAUDE.md when conventions span the codebase.
  • 3.4 Plan mode vs direct execution. Plan for large / multi-approach / architectural (microservices, 45+ file migrations); direct for a clear single-file fix with a stack trace. Explore subagent isolates discovery. Combine plan then execute.
  • 3.5 Iterative refinement. Concrete I/O examples; test-driven iteration; the interview pattern (Claude asks questions to surface design); interdependent issues in one message vs independent sequentially.
  • 3.6 CI/CD integration. -p/--print non-interactive; --output-format json + --json-schema; CLAUDE.md gives CI context; session context isolation (independent instance reviews better); report only new/unfixed; include existing tests in context.

Domain 4 - Prompt Engineering & Structured Output (20%) - 6 task areas

  • 4.1 Explicit criteria. Concrete categorical criteria beat vague ("flag comments only when they contradict code"). False positives erode trust. Define report vs ignore; severity with code examples.
  • 4.2 Few-shot. Most effective for consistent, actionable output; 2-4 targeted examples + rationale; demonstrate format (location/issue/severity/fix); distinguish acceptable vs real issues; reduces extraction hallucination.
  • 4.3 Structured output tool_use + JSON Schema. Most reliable schema-conformant output; kills syntax errors, not semantic. tool_choice auto/any/forced. Required vs optional/nullable to avoid fabrication. Enums + "other"/"unclear" + detail string.
  • 4.4 Validation, retries, feedback. Retry-with-error-feedback (original doc + bad extraction + exact errors). Retries useless when info is absent. detected_pattern field. Self-correction: extract calculated_total vs stated_total.
  • 4.5 Batch processing. Message Batches API: 50% cheaper, up to 24h, no SLA. Non-blocking only (overnight/audits), not pre-merge. No multi-turn tool calling in one request. custom_id correlation; cadence planning; re-submit only failed by custom_id.
  • 4.6 Multi-instance / multi-pass review. Self-review is weak (retains its reasoning). Independent instance finds subtle issues. Per-file + cross-file integration pass. Verification with self-rated confidence for calibrated routing.

Domain 5 - Context Management & Reliability (15%) - 6 task areas

  • 5.1 Preserve critical info. Summarization loses numbers/percentages/dates. Lost-in-the-middle (start + end processed, middle missed). Tool outputs accumulate (40+ fields when 5 needed). Send full history. "Case facts" block; trim tool outputs; key findings at the start with headings; subagents include metadata.
  • 5.2 Escalation & ambiguity. Triggers: explicit human request, policy gap, no progress. Immediate vs attempt-to-resolve. Sentiment + confidence self-ratings are unreliable. Multiple matches: ask for identifiers, do not guess. Explicit criteria + few-shot in the system prompt.
  • 5.3 Error propagation (multi-agent). Structured error context (failure type, query, partial results, alternatives). Access failure vs valid empty result. Generic statuses hide context. Silent suppression and full-abort are both anti-patterns. Local recovery for transient; propagate non-recoverable with partial results. Coverage annotations in synthesis.
  • 5.4 Large-codebase context. Degradation in long sessions (model cites "typical patterns" instead of specific classes). Scratchpad files. Delegate to subagents to isolate discovery. Structured state persistence for crash recovery. /compact.
  • 5.5 Human oversight & confidence calibration. Aggregate metrics mask per-type/field failures. Stratified random sampling. Field-level confidence with labeled validation sets. Validate accuracy by doc type/field before automating. Route low-confidence / ambiguous-source items to human review.
  • 5.6 Information provenance & uncertainty in synthesis. Summarization can lose claim-source mappings. Require subagents to return source URLs or document names, relevant excerpts, and publication or collection dates. Preserve conflicts from credible sources with attribution instead of choosing one value. Separate well-established findings from contested ones, and render financial data as tables, news as prose, and technical findings as structured lists when that better preserves meaning.

The 12 OFFICIAL sample questions (verbatim from the guide, Jun 17)

These are the real sample questions printed in the official guide. They are the single best model of the exam's style: a production scenario, 4 options, pick the BEST. Use them as the gold standard when writing mock questions, and have Shela attempt them cold. The right answer and the one-line lesson are given.

Customer Support Resolution Agent

  1. Agent skips get_customer in 12% of cases and misidentifies accounts. Fix? A: add a programmatic prerequisite that blocks lookup_order/process_refund until get_customer returns a verified ID. Lesson: critical business logic needs deterministic enforcement (hooks/preconditions), not prompt instructions (B, C) or tool-availability changes (D).
  2. Agent calls get_customer for order questions; both tools have minimal near-identical descriptions. Best FIRST step? B: expand each tool's description (input formats, example queries, edge cases, boundaries). Lesson: tool descriptions are the primary selection mechanism; fix the descriptions, do not add few-shot (A), a routing classifier (C), or consolidate tools (D) as a first step.
  3. 55% first-contact resolution; agent escalates easy cases and tries to autonomously handle hard policy-exception ones. Fix escalation? A: add explicit escalation criteria + few-shot examples to the system prompt. Lesson: explicit criteria beat self-reported confidence (B, poorly calibrated), a trained classifier (C, over-engineered), and sentiment detection (D, sentiment does not equal complexity).

Code Generation with Claude Code

  1. Custom /review slash command available to every dev on clone/pull. Where? A: .claude/commands/ in the project repo (version-controlled, shared). Not user-level (B), not CLAUDE.md (C), not a non-existent config.json (D).
  2. Restructure a monolith into microservices, dozens of files, boundary decisions. Approach? A: plan mode to explore, understand dependencies, design before changing. Lesson: plan mode is for large-scale/architectural/multi-file work; direct execution (B, C, D) risks costly rework.
  3. Codebase with distinct conventions, test files spread throughout, want auto-applied conventions by file type. A: .claude/rules/ with YAML frontmatter glob patterns (e.g. */.test.tsx). Beats a monolithic CLAUDE.md (B), skills needing manual invocation (C), or directory-bound CLAUDE.md files (D).

Multi-Agent Research System

  1. Reports cover only visual arts, missing music/writing/film; coordinator logs show it decomposed into only visual-art subtasks. Root cause? B: the coordinator's task decomposition is too narrow. Lesson: read the logs; the subagents did their assigned jobs correctly, so do not blame downstream agents (A, C, D).
  2. Web search subagent times out; design failure flow to the coordinator. A: return structured error context (failure type, attempted query, partial results, alternatives). Lesson: structured errors enable recovery; generic statuses (B), suppressing as success (C), and aborting the whole workflow (D) are all anti-patterns.
  3. Synthesis agent needs frequent fact-checks (85% simple, 15% deep), causing 2-3 round trips and +40% latency. A: give synthesis a scoped verify_fact tool for simple lookups; keep complex verifications routed through the coordinator. Lesson: least privilege + scoped cross-role tool for the common case; do not batch (B), over-provision all web tools (C), or speculatively cache (D).

Claude Code for Continuous Integration

  1. Pipeline runs claude "Analyze this PR" and hangs on interactive input. A: add the -p (--print) flag. Lesson: -p is the documented non-interactive mode; the others are non-existent flags/env vars or Unix workarounds.
  2. Two workflows (a blocking pre-merge check, an overnight report); manager wants both on the Batch API for 50% savings. A: batch for the overnight report only; keep real-time for the blocking pre-merge check. Lesson: Batch API is up to 24h with no latency SLA, fine for overnight, wrong for blocking; match the API to the latency need.
  3. PR touches 14 files; single-pass review gives inconsistent, contradictory feedback. A: split into focused passes (per-file local analysis, then a separate cross-file integration pass). Lesson: file-by-file avoids attention dilution; a bigger model (C) does not fix attention, and consensus-of-3 (D) suppresses real intermittent bugs.

The pattern across all 12: the winning answer is almost always the boring, well-bounded, deterministic-where-it-matters choice. The distractors are the four losing shapes: over-engineering, ignoring failure, trusting the model to behave (sentiment/confidence), or full autonomy where a gate belongs. None ask "write the correct code."

The official Exam Preparation Recommendations (Jun 17)

The guide's own prep advice is heavily HANDS-ON. This matters for how to coach Shela: the guide assumes judgment is earned by building.

  1. Build an agent with the Claude Agent SDK (full agentic loop, tool calling, error handling, session management, subagents).
  2. Configure Claude Code for a real project (CLAUDE.md hierarchy, .claude/rules/, skills with frontmatter, MCP server).
  3. Design and test MCP tools (differentiated descriptions, structured error responses, ambiguous-request testing).
  4. Build a structured data extraction pipeline (tool_use + JSON schema, validation-retry, optional/nullable fields, Batch API).
  5. Practice prompt engineering (few-shot for ambiguity, explicit review criteria, multi-pass review).
  6. Study context management (structured facts from verbose outputs, scratchpad files, subagent delegation).
  7. Review escalation / human-in-the-loop (when to escalate vs resolve, confidence-based routing).
  8. Complete the official practice exam before sitting the real one.

There are also 4 detailed hands-on "Preparation Exercises" in the guide (build a multi-tool agent with escalation; configure Claude Code for a team; build an extraction pipeline; design and debug a multi-agent research pipeline). These map directly to the 6 scenarios and are the exact "build-and-break" work to do on the slow pass.
