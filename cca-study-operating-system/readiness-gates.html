CCA Readiness Gates

Purpose: Define what "ready for the CCA-F exam" means. Do not rely on reassurance or course completion.

Gate 1: Guide Coverage

Shela is not ready until:

  • all 30 task rows in guide-mastery-matrix.md are at least Practiced;
  • at least 25 of 30 task rows are Passed;
  • no Domain 1 row is below Practiced;
  • no row marked Needs review has gone unaddressed for more than 3 study sessions.

Gate 2: Vocabulary

Shela must answer at least 45 of 50 vocabulary prompts correctly without notes.

Required vocabulary groups:

  • API and stateless requests
  • agent, agent loop, coordinator, subagent
  • tool, tool schema, structured error
  • MCP tool, MCP resource, MCP prompt, MCP server
  • JSON, JSON schema, optional/nullable fields
  • context window, summarization, case facts, source mapping
  • Claude Code, CLAUDE.md, slash commands, skills, plan mode
  • hooks, escalation, human review, confidence calibration

Gate 3: Scenario Judgment

Shela must explain all 6 official scenarios without notes:

  1. What is being built.
  2. Which domains are reinforced.
  3. Which tools or Claude surface are involved.
  4. What can fail.
  5. Which answer choices are traps.
  6. What the safe, boring design instinct is.

Passing threshold: 5 of 6 scenarios strong, and the weak one immediately reviewed.

Gate 4: Mixed Mock MCQs

Run 30 mixed mock MCQs across all domains.

Passing threshold:

  • 24/30 or higher overall;
  • no domain below 70%;
  • every missed question has a written wrong-answer explanation.

The trainer must label these as practice questions, not official exam questions.

Gate 5: Explain-It-Back

Shela must explain these without notes:

  • why subagents need explicit context;
  • why schema guarantees shape, not truth;
  • why hooks enforce better than prompts;
  • why sentiment/confidence are bad escalation triggers;
  • why plan mode differs from extended thinking;
  • why Batch API is wrong for blocking CI;
  • why exact question count/time limit should not be memorized from community sources.

Passing threshold: plain explanation plus a concrete example for each.

Gate 6: Out-of-Scope Recognition

Shela must identify at least 12 out-of-scope topics from the official guide and say why they should not be over-studied.

Passing threshold: 12/15 correct in a mixed in-scope/out-of-scope drill.

Gate 7: Hands-On Design

Shela must complete at least one design exercise:

  • a coordinator;
  • two subagents;
  • one MCP-style tool with a clear schema;
  • structured error handling;
  • JSON schema final output;
  • one failure case and one escalation path.

This can be a design document or small implementation. The goal is design judgment, not production coding.

Final Readiness Rule

The trainer can say "ready" only when Gates 1-6 pass and Gate 7 is at least designed. If any gate fails, the trainer must name the smallest next fix instead of reassuring.
