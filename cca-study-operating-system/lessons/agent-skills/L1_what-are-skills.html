Lesson 1 - What are skills?

Learning objectives (from the course)

  • Define what Claude Code skills are and how they work
  • Explain where skills live (personal vs. project directories)
  • Distinguish between skills, CLAUDE.md, and slash commands
  • Identify scenarios where skills are the right customization tool

The lesson in plain English

A skill is a markdown file that teaches Claude how to do something once, so you stop re-explaining it every time. More fully: skills are folders of instructions, scripts, and resources that Claude can discover and use to do tasks more accurately. Each skill lives in a SKILL.md file whose frontmatter (top block) holds a name and a description.

The description is the matching mechanism. When you ask Claude to "review this PR," it compares your request against the descriptions of all available skills and activates the ones that match. That is the whole trigger model.

Where they live decides who gets them:

  • Personal → ~/.claude/skills/ - follows you across every project (your commit style, your doc format).
  • Project → .claude/skills/ in the repo root - anyone who clones the repo gets them automatically (team standards, brand guidelines).

How skills differ from the neighbors:

  • CLAUDE.md loads into every conversation (always-on). Skills load on demand - only the name + description sit in context until a match, so they don't bloat your context window.
  • Slash commands require you to type them. Skills activate automatically when Claude recognizes the situation.

Rule of thumb: if you find yourself explaining the same thing to Claude repeatedly, that's a skill waiting to be written.

Key takeaways (verbatim from course companion)

  • Skills are folders of instructions that Claude Code can discover and use to handle tasks more accurately. Each skill lives in a SKILL.md file with a name and description in its frontmatter
  • Claude uses the description to match skills to requests. When you ask Claude to do something, it compares your request against available skill descriptions and activates the ones that match
  • Personal skills go in ~/.claude/skills and follow you across all projects. Project skills go in .claude/skills inside a repository and are shared with anyone who clones it
  • Skills load on demand - unlike CLAUDE.md (which loads into every conversation) or slash commands (which require explicit invocation), skills activate automatically when Claude recognizes the situation
  • If you find yourself explaining the same thing to Claude repeatedly, that's a skill waiting to be written

Exam watch-outs (CCA-F, Domain 3)

  • Project vs. user/personal split is a classic trap. Team skill at user-level = teammates can't see it. Shared = in the repo (.claude/skills/), under version control.
  • The description is what Claude matches on - same design instinct as tool descriptions in Domain 2. Vague description = no trigger.
  • Know the three-way contrast cold: CLAUDE.md = always-on, skills = on-demand/auto, slash commands = on-demand/manual.

Terms introduced

  • Skill = a folder + SKILL.md teaching Claude a task once, applied automatically when relevant.
  • SKILL.md = the file defining a skill; frontmatter (name, description) + instructions below.
  • description (frontmatter) = the text Claude uses to decide whether the skill matches a request.
  • Personal skill = ~/.claude/skills/, follows you across projects.
  • Project skill = .claude/skills/ in repo, shared with everyone who clones.
