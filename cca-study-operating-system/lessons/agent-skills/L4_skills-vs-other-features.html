Lesson 4 - Skills vs. other Claude Code features

Learning objectives (from the course)

  • Compare skills to CLAUDE.md, subagents, hooks, and MCP servers
  • Choose the right Claude Code customization feature for a given use case
  • Design a complementary setup that combines multiple features effectively

The lesson in plain English

Five customization options, each solving a different problem. Picking the wrong one creates needless complexity. The distinctions (this is the heart of the exam's "choose the right tool" questions):

FeatureTrigger modelContextBest for
CLAUDE.mdAlways-on (every conversation)Joins every chatProject-wide standards, constraints ("never modify the DB schema"), framework/style prefs
SkillsRequest-driven (auto on match)Joins current conversationTask-specific expertise; knowledge only sometimes relevant; procedures that would clutter every chat
SubagentsYou delegateIsolated/separate contextDelegating a task; needing different tool access; isolation from the main context
HooksEvent-driven (file save, tool call)n/aRun-on-every-save ops, validation before specific tool calls, automated side effects
MCP serversExternal integrationn/aExternal tools and integrations (a different category entirely)

The punchline: each handles its own specialty - combine them, don't force everything into skills. A typical setup: CLAUDE.md for always-on standards + skills for task-specific expertise + hooks for automated ops + subagents for isolated delegation + MCP for external tools.

Key takeaways (verbatim from course companion)

  • CLAUDE.md loads into every conversation and is best for always-on project standards. Skills load on demand and are best for task-specific expertise
  • Subagents run in isolated execution contexts - use them for delegated work. Skills add knowledge to your current conversation
  • Hooks are event-driven (fire on file saves, tool calls). Skills are request-driven (activate based on what you're asking)
  • MCP servers provide external tools and integrations - a different category entirely from skills
  • Each feature handles its own specialty - combine them rather than forcing everything into one approach

Exam watch-outs (CCA-F) - this lesson is high-yield

  • The two axes that separate them: trigger (always-on vs. request vs. event vs. delegated) and context (shared current conversation vs. isolated). Skills = request-driven + shared; subagents = delegated + isolated; hooks = event-driven.
  • Hooks vs. prompts/skills for guarantees (Domain 1, task 1.5): hooks are deterministic (code, fire on events); skills/prompts are probabilistic guidance. For financial/legal/safety guarantees, the exam wants hooks, not a skill.
  • Subagents are isolated and inherit nothing - ties directly to Domain 1 orchestration. (Lesson 5 adds: they don't even see skills unless explicitly listed.)
  • Over-engineering trap: "force everything into one approach" is the wrong answer; the boring "combine the right feature for each job" is right.

Terms introduced

  • Always-on vs. on-demand = CLAUDE.md is always loaded; skills load only when matched.
  • Event-driven vs. request-driven = hooks fire on events (saves, tool calls); skills fire on what you ask.
  • Isolated execution context = a subagent's separate context that doesn't share the main conversation.
