Lesson 5 - Sharing skills

Learning objectives (from the course)

  • Share skills with your team by committing them to a Git repository
  • Distribute skills across projects through plugins and marketplaces
  • Deploy skills organization-wide using enterprise managed settings
  • Configure custom subagents to use specific skills

The lesson in plain English

A skill only you use is helpful; the same skill shared across the team standardizes everyone. Three distribution methods:

  1. Commit to your repo (simplest) - put skills in .claude/skills/; anyone who clones gets them, no install. Push updates → everyone gets them on next pull. Good for team coding standards, project workflows, codebase-specific skills.
  2. Plugins - package skills (a skills/ dir mirroring the .claude structure, each skill its own folder + SKILL.md) and distribute via a marketplace. Others install into their own Claude Code. Best for skills that aren't too project-specific and help the broader community.
  3. Enterprise managed settings - admins deploy org-wide. Enterprise skills take highest priority, overriding personal/project/plugin skills of the same name. For mandatory standards, security, compliance - keyword "must."

The gotcha people miss: subagents don't automatically see your skills. A delegated subagent starts with a fresh, clean context.

  • Built-in agents (Explorer, Plan, Verify) can't access skills at all.
  • Custom subagents can - but only when you explicitly list them in the agent's frontmatter skills: field (in a .claude/agents/*.md file).
  • For subagents, skills load when the subagent starts (not on demand like the main conversation), so only list skills always relevant to that agent's purpose.
