Lesson 3 - Configuration and multi-file skills

Learning objectives (from the course)

  • Configure advanced skill metadata fields including allowed-tools and model
  • Write effective skill descriptions that reliably trigger on the right requests
  • Use allowed-tools to restrict what Claude can do when a skill is active
  • Organize complex skills using progressive disclosure and multi-file structures

The lesson in plain English

The agentskills.io open standard defines the SKILL.md frontmatter fields:

  • name (required) - lowercase letters, numbers, hyphens only; max 64 chars; should match the directory name.
  • description (required) - max 1,024 chars; the most important field because Claude matches on it. A good description answers two questions: what does the skill do? and when should Claude use it? If it isn't triggering, add keywords matching how you actually phrase requests.
  • allowed-tools (optional) - restricts which tools Claude may use while the skill is active. Omit it and the skill uses Claude's normal permission model. Useful for read-only / security-sensitive workflows (e.g. Read, Grep, Glob only - no editing, writing, or bash).
  • model (optional) - specifies which Claude model the skill should use.

Progressive disclosure (the big idea): skills share Claude's context window with your conversation, and loading one pulls its SKILL.md into context. Don't cram everything into one giant file. Keep SKILL.md under 500 lines with essential instructions, and link to supporting files that Claude reads only when needed. The standard suggests organizing a skill directory with:

  • scripts/ - executable code
  • references/ - additional documentation
  • assets/ - images, templates, data files

It's "a table of contents in the context window rather than the whole document." Claude reads references/architecture.md only when someone asks about system design.

Scripts run without loading their contents into context - the script executes and only its output consumes tokens. Tell Claude to run the script, not read it. Best for environment validation, consistent data transformations, and operations more reliable as tested code than as generated code.

Key takeaways (verbatim from course companion)

  • name and description are required - allowed-tools and model are optional but powerful additions

  • Progressive disclosure: keep SKILL.md under 500 lines and link to supporting files (references, scripts, assets) that Claude reads only when needed
  • Scripts execute without loading their contents into context - only the output consumes tokens, keeping context efficient

Exam watch-outs (CCA-F)

  • allowed-tools = a guardrail, not a security boundary. It scopes tools when the skill is active (Domain 2, task 2.3 tool allocation). Real security is enforced in code (see MCP lessons).
  • Progressive disclosure is a context-management answer (Domain 5): the "right" choice when a skill's reference material is large is link it, load on demand, not "put it all in SKILL.md."
  • Run-don't-read scripts: only output costs tokens. A likely "how do you keep context efficient" item.
  • Field constraints worth remembering: name ≤64 chars / matches dir; description ≤1,024 chars and is the matching field.

Terms introduced

  • allowed-tools = optional frontmatter field restricting tools while the skill is active.
  • model = optional field naming which Claude model the skill uses.
  • Progressive disclosure = keep SKILL.md lean (<500 lines), link out to scripts/references/assets loaded only when needed.
  • scripts/ references/ assets/ = the open-standard sub-folders of a multi-file skill.
